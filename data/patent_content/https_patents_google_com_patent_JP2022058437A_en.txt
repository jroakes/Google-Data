本発明は、ネットワーク化環境における遅延２要素認証に関する。 The present invention relates to delayed two-factor authentication in a networked environment.
コンピューティングデバイスが、ネットワーク化クライアントデバイスにデジタルコンテンツを提供することができる。コンピューティングデバイスは、ネットワーク化クライアントデバイスにコンテンツを送信する前に認証または認可を必要とし得る。音声駆動インタフェースなど、限られたユーザインタフェースを持つクライアントデバイスを認証することは困難であり得る。 Computing devices can provide digital content to networked client devices. Computing devices may require authentication or authorization before sending content to networked client devices. It can be difficult to authenticate a client device with a limited user interface, such as a voice driven interface.
本開示の少なくとも1つの態様によれば、コンピューティングデバイスを認証するシステムがデータ処理システムを含むことができる。データ処理システムは、自然言語プロセッサコンポーネント、コンテンツ選択コンポーネントおよび認可コンポーネントを実行できる。データ処理システムは、第1のクライアントデバイスにおいてセンサによって検出される入力オーディオ信号を受信できる。自然言語プロセッサコンポーネントは、入力オーディオ信号を構文解析して、入力オーディオ信号中に第1のデジタルコンポーネント要求を識別できる。コンテンツ選択コンポーネントは、第1のクライアントデバイスに第1のデジタルコンポーネントの第1の部分を送信できる。認可コンポーネントは、第1のクライアントデバイスへのデジタルコンポーネントの第2の部分の送信を認可するために第2のクライアントデバイスに認可要求を送信できる。認可要求は、第1のクライアントデバイスに第1のデジタルコンポーネントの第1の部分を送信したことに少なくとも基づいて第2のクライアントデバイスに送信できる。認可コンポーネントは、第2のクライアントデバイスから、認可要求に応答した確認メッセージを受信できる。コンテンツ選択コンポーネントは、第2のクライアントデバイスから確認メッセージを受信したことに少なくとも基づいて第1のクライアントデバイスにデジタルコンポーネントの第2の部分を送信できる。 According to at least one aspect of the present disclosure, a system that authenticates a computing device can include a data processing system. The data processing system can execute a natural language processor component, a content selection component, and an authorization component. The data processing system can receive the input audio signal detected by the sensor in the first client device. The natural language processor component can parse the input audio signal to identify the first digital component requirement in the input audio signal. The content selection component can send the first part of the first digital component to the first client device. The authorization component can send an authorization request to the second client device to authorize the transmission of the second part of the digital component to the first client device. The authorization request can be sent to the second client device based on at least sending the first part of the first digital component to the first client device. The authorization component can receive a confirmation message in response to the authorization request from the second client device. The content selection component can send a second part of the digital component to the first client device, at least based on receiving a confirmation message from the second client device.

The present disclosure generally covers the authentication of audio-based interactive devices in interconnect networks. For example, a digital assistant component can receive audio input that can contain speech. The speech can include a request for content. The Digital Assistant component can request content from third parties in response to audio input. Delivery of content to digital assistant components may require authorization. It can be difficult to authenticate devices that use voice-based interfaces. Some inputs, such as credit card numbers or passwords that end users do not want to speak out due to privacy and security concerns, can be difficult or disturbing to provide through a voice-based interface. In addition, voice-based interfaces can result in excessive computational resources, network bandwidth, or remote procedure calls, such as processor and memory resources, as a result of generating a conversational interface that authorizes content delivery. It can be difficult or inefficient to authorize a device with. In addition, as a result of the need for immediate authorization through the non-voice-based interface of the second device, the end user usually gives up the request rather than switching to interface through the second device. Network resources can be wasted.

Digital components can be provided through subscription-based services (eg, streaming music services) or may require payment authorization (eg, purchasing songs). Delivery of digital components to digital assistants may require authorization in a second, non-voice driven client device. This system of technical solutions can include state machines. The state machine can track the availability of services that provide digital components (to the digital assistant) as well as the authentication status of the digital assistant.

The system can send authorization requests to a second client device associated with the digital assistant. The system can provide the second client device with a predetermined amount of time to respond to an authorization request. During this predetermined amount of time, the digital assistant can render or present the requested digital component. Once an authorization request is received, the second device can indicate a notification with confirmation and denial options, or the notification may open a dialog with those options. The second client device can automatically approve the authorization request based on the local device state, such as whether the second client device is within a given distance from the digital assistant. Upon confirmation of the dialog (or other explicit involvement in the authorization request) by the end user, the second client device may send an authentication token and a unique action identifier to the system. The system can use an authentication token and a unique action identifier to transform the digital assistant's state machine into an authorized state.
