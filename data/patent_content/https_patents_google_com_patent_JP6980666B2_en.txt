Description
本発明は、オンデマンドの位置訪問コンバージョンメトリックに関する。 The present invention relates to an on-demand location visit conversion metric.
ウェブベースの検索は、店舗、レストラン、または他の事業所で提供される製品およびサービスに関する情報を含む、様々な有益な情報をユーザに提供する。典型的なブラウザベースのアプリケーションでは、ユーザは、特定の位置に関係する、または関心のあるエリア内の製品またはサービスに関する情報を要求する場合がある。ユーザがこの情報を実際に利用したかどうかを判断することが重要である。いくつかのシステムは、クリックスルーレート(CTR)を評価することによってこれを推定する。CTRは、リンクが提供されたユーザの総数に対する特定の情報の1つのリンクをクリックしたユーザの数の比率である。他の同様のメトリックおよびアナリティクスが使用されてもよいが、多くの場合、ユーザが提供された情報に実際に頼っているかどうかに関する概算推定を提供するだけである。 Web-based searches provide users with a variety of useful information, including information about products and services offered at stores, restaurants, or other establishments. In a typical browser-based application, a user may request information about a product or service in an area that is relevant or of interest to a particular location. It is important to determine if the user has actually used this information. Some systems estimate this by assessing click-through rate (CTR). CTR is the ratio of the number of users who clicked a link for a particular piece of information to the total number of users for whom the link was provided. Other similar metrics and analytics may be used, but often only provide a rough estimate as to whether the user is actually relying on the information provided.
ユーザが提供された情報を利用するかどうかを判断するために、より効果的な方法を実装することが重要である。これは、ユーザが情報を利用したかどうかを知ることで、検索結果がどのように関連しているかをプロバイダに伝えるためである。将来的な結果をユーザにとってより意味のある方法で潜在的に提示する方法をも提案する可能性がある。 It is important to implement more effective methods to determine whether the user will use the information provided. This is to tell the provider how the search results are relevant by knowing if the user has used the information. It may also suggest ways to potentially present future results in a way that is more meaningful to the user.
いくつかの場合には、ユーザが自分の位置履歴に関する情報をシステムが受信できるようにすると、検索結果など受信された情報のコンバージョンレートを評価することが可能である。ここでは、CTRメトリクスを強化するために、検索結果とクリックとの位置履歴を相互参照することができる。たとえば、ユーザが特定の会社に関連付けられた検索結果をクリックし、さらにその会社を訪問した場合、これは、コンバージョンとしてカウントされる。しかしながら、これは、ユーザの検索履歴とデバイスの位置履歴に関する許可を有効にした極めて少ないサブセットのユーザに適用される場合がある。また、その少ないサブセットは、関連の統計を提供するのに十分ではない場合がある。 In some cases, allowing the system to receive information about its location history allows the user to evaluate the conversion rate of the received information, such as search results. Here you can cross-reference the location history of search results and clicks to enhance CTR metrics. For example, if a user clicks on a search result associated with a particular company and then visits that company, this counts as a conversion. However, this may apply to a very small subset of users who have enabled permissions on the user's search history and device location history. Also, its small subset may not be sufficient to provide relevant statistics.

Accordingly, the technique determines if the user has subsequently visited a location or area associated with the information provided. This technique does not need to collect or export individual location information for each user over a period of time. Rather, wireless station information is provided to client devices along with location-based search results. The client device can then determine if a particular wireless station is subsequently visible, in other words, within communication range. This can be done, for example, by matching the observed WiFi ID with the stored WiFi ID received as part of the received wireless station information. One advantage is that this method is very inexpensive in terms of computational or processing power. It also does not use any extra power, so it does not impose any burden on the client device. Any match between the observed and stored IDs can be stored locally on the client device, after which some indicators associated with such matches are returned to the network. In this case, the anonymized result may be provided to a network element such as a web server. As a result, the system can accurately track conversions and search quality, which improves the user experience and provides a more efficient search results process. 概要
システムは、Bluetoothトランスミッタ、パーソナルエリアネットワークデバイス、たとえば会社および他の関心のあるポイントに関連付けられたIEEE802.15.4準拠デバイスなど(総称して、ワイヤレス局)を含む、WiFiアクセスポイントまたは他のタイプの基地局またはアドホック構成に関する情報を保持する。情報は、物理的位置、会社の名前など、対応する位置を有するワイヤレス局IDを含む。この情報は、ネットワーク側の、たとえば1つまたは複数のサーバのデータベースに記憶される。
Overview The system is a WiFi access point or other type, including Bluetooth transmitters, personal area network devices, such as IEEE 802.15.4 compliant devices associated with companies and other points of interest (collectively, wireless stations). Holds information about base stations or ad hoc configurations. The information includes a wireless station ID with a corresponding location, such as physical location, company name, and so on. This information is stored in the database on the network side, for example one or more servers.
クライアント側検索クエリまたは他の要求をクライアントデバイスから受信すると、ネットワーク側は、要求に関連する位置を識別する。識別された位置を使用して、1つまたは複数のワイヤレス局のセットが選択される。識別された位置に関連付けられた1つまたは複数のWiFiアクセスポイントに関する情報は、クエリへの応答の一部としてクライアントデバイスに送信される。一例では、これは、検索クエリに関連付けられた店舗または関心のあるポイントに位置するWiFiアクセスポイントのIDを含んでもよい。店舗が複数のアクセスポイントまたは他のワイヤレス局を有する場合、ワイヤレス局のIDの一部または全部にフラグが立てられてもよい。オプションで、他の近くにあるワイヤレス局のIDにもフラグが立てられてもよい。 Upon receiving a client-side search query or other request from a client device, the network side identifies the location associated with the request. A set of one or more wireless stations is selected using the identified location. Information about one or more WiFi access points associated with the identified location is sent to the client device as part of the response to the query. In one example, this may include the ID of a WiFi access point located at the store or point of interest associated with the search query. If the store has multiple access points or other wireless stations, some or all of the wireless station IDs may be flagged. Optionally, the IDs of other nearby wireless stations may also be flagged.
検索クエリに応答する位置関連コンテンツもまた取得される。これは、識別された位置にある会社に関連付けられた製品またはサービスに関する情報を含んでいる可能性がある。また、たとえば販売、提供、または販売促進に関する広告関連情報も含んでもよい。 Location-related content that responds to search queries is also retrieved. It may contain information about the product or service associated with the company in the identified location. It may also include, for example, advertising-related information about a sale, offer, or promotion.
検索クエリに対する応答は、選択されたワイヤレス局IDに関する情報を含む。一例では、クエリ応答およびワイヤレス局IDのセットは、1つまたは複数のデータパケットなどで一緒に提供される。別の例では、クエリ応答およびワイヤレス局IDのセットは、別々に提供されてもよい。 The response to the search query contains information about the selected wireless station ID. In one example, the query response and set of wireless station IDs are provided together, such as in one or more data packets. In another example, the query response and the set of wireless station IDs may be provided separately.
クライアント側では、デバイスは、ひとたび応答を受信すると、検索結果または他の情報に作用することができる。クライアントデバイスはまた、受信されたワイヤレス局IDをメモリに記憶する。たとえば、クエリまたは他の要求を発行したアプリケーションまたは他のサービスは、関連する検索結果をクライアントデバイスのユーザに提供してもよい。クライアントデバイスはまた、受信されたワイヤレス局IDを評価する。特に、クライアントデバイスの1つまたは複数のプロセッサは、記憶されたIDをクライアントデバイスのワイヤレスレシーバ(またはトランシーバ)によって取得されたスキャンされた情報と比較してもよい。例として、典型的な動作の間、クライアントデバイスは、利用可能なWiFiアクセスポイントおよび/または他のワイヤレス局を日常的にスキャンしてもよい。可視である(たとえば、通信範囲内にある)任意のそのようなワイヤレス局のIDは、記憶されたIDと比較される。 On the client side, the device can interact with search results or other information once it receives a response. The client device also stores the received wireless station ID in memory. For example, an application or other service that issued a query or other request may provide relevant search results to users on the client device. The client device also evaluates the received wireless station ID. In particular, one or more processors on the client device may compare the stored ID to the scanned information obtained by the client device's wireless receiver (or transceiver). As an example, during typical operation, the client device may routinely scan for available WiFi access points and / or other wireless stations. The ID of any such wireless station that is visible (eg, within range) is compared to the stored ID.
一致がある場合、その情報はクライアントデバイスによって記憶される。特に、モバイルデバイスは、一致があるかどうかを示すログ情報を作成してもよい。たとえば、ログは、ワイヤレス局IDおよび「訪問」インジケータを含んでもよい。ワイヤレス局がいつ「訪問されたか」に関するタイミング情報を含む必要はない。または、ログデータは、1がtrueに等しく(IDの一致)、0がfalseに等しい(IDの一致なし)の単純なブーリアンインジケータを含んでもよい。または、ログはカウンタとして機能し、この場合、一致の総数は段階的に増加する。ログデータの形式に関係なく、クライアント固有の情報は含まれておらず、したがって、ログの匿名化が行われる。 If there is a match, that information is stored by the client device. In particular, mobile devices may create log information indicating whether there is a match. For example, the log may include a wireless station ID and a "visit" indicator. It is not necessary to include timing information about when the wireless station was "visited". Alternatively, the log data may contain a simple Boolean indicator where 1 is equal to true (ID match) and 0 is equal to false (no ID match). Alternatively, the log acts as a counter, in which case the total number of matches is incremented. Regardless of the format of the log data, it does not contain any client-specific information and therefore the log is anonymized.
1つのシナリオでは、ログ情報は、ネットワークに送信される前に、所与の時間期間、たとえば1時間、1日などにわたって蓄積されてもよい。別のシナリオでは、ログ情報は、一致の判定後直ちに送信される。この情報は、いったんネットワークで受信されると、アグリゲートされ、可能なコンバージョンレートを評価するために使用される。図1は、上記の説明による例示的なクエリおよび応答プロセスを示す。技術のこれらおよび他の態様について、以下でさらに説明される。 In one scenario, log information may be accumulated over a given time period, eg, an hour, a day, etc., before being sent to the network. In another scenario, the log information is sent immediately after the match is determined. Once received over the network, this information is aggregated and used to assess possible conversion rates. FIG. 1 shows an exemplary query and response process as described above. Exemplary Methods and Use Cases As mentioned above, the features of this technique can be implemented in different use cases. An exemplary use case is described below. For example, in one use case, a user may request information via a mobile app or service on a client device. As an example, the information can be about points of interest, such as restaurants, coffee shops, car dealerships, museums, and so on. Other use cases may involve online shopping or other orders. Further examples may include a map or direction query. Nevertheless, the technology does not depend on queries and search results.
クエリ、またはクエリを管理するアプリケーションもしくはサービスのタイプにかかわらず、クエリに基づく要求は、クライアントデバイスの通信モジュールを介してネットワークに送信される。ウェブサーバの通信モジュールによって受信されると、ウェブサーバの処理システムによってクエリが分析され、クエリに関連する位置が識別される。この識別に基づいて、ウェブサーバは、クライアントデバイスによる検出の候補として、その位置またはその近くの1つまたは複数のワイヤレス局を選択する。 Regardless of the type of query or application or service that manages the query, query-based requests are sent to the network through the communication module of the client device. When received by the web server's communication module, the web server's processing system analyzes the query and identifies the location associated with the query. Based on this identification, the web server selects one or more wireless stations at or near its location as candidates for detection by the client device.
特に、クライアントデバイスが識別された位置に(またはその近くに)あるとき、クライアントデバイスが見る1つまたは複数のワイヤレス局が選択される。ワイヤレス局は、WiFiアクセスポイントなどを含んでもよい。一例では、そのような局の異なるタイプおよび組合せが選択されてもよい。 In particular, when the client device is in (or near) the identified location, one or more wireless stations that the client device sees are selected. The wireless station may include a WiFi access point and the like. In one example, different types and combinations of such stations may be selected.

The
ログ情報は、たとえば最初の検索クエリが提出されたウェブサーバに戻るなど、ネットワークへの送信前に、たとえば1時間、1日など所与の時間期間にわたって蓄積されてもよい。あるいはログ情報は、必要な場合、直ちに送信されてもよい。ひとたび送信されると、ログ情報は、クライアントデバイスのメモリから削除されてもよい。 Log information may be accumulated over a given period of time, for example an hour, a day, etc., before being sent to the network, for example returning to the web server where the first search query was submitted. Alternatively, the log information may be sent immediately if necessary. Once sent, the log information may be deleted from the memory of the client device.
ログ情報がクライアントデバイスによってネットワークに送信されるとき、ユーザIDまたは他のクライアント識別可能な情報はログに含まれないことに留意することが重要である。追加されたセキュリティ層として、ログデータに対して統計的ランダム化が実行されてもよい。統計的ランダム化の1つの手法は、「訪問」インジケータを「非訪問」ステータスにランダムに変更すること、およびその逆を含む。これは、所定の量のログ情報に対して実行されてもよい。単に例として、クライアントデバイスの処理システムは、ログ情報の10%に対してそのような統計的ランダム化を実行してもよい。別の例では、ログ情報の少なくとも5%がランダム化される。また、さらなる一例では、ログ情報の25%以下がランダム化される。 It is important to note that when log information is sent by the client device to the network, the user ID or other client identifiable information is not included in the log. As an added layer of security, log data may be statistically randomized. One technique of statistical randomization involves randomly changing the "visit" indicator to a "non-visit" status, and vice versa. This may be done for a given amount of log information. As an example only, the processing system of the client device may perform such statistical randomization on 10% of the log information. In another example, at least 5% of the log information is randomized. In another example, less than 25% of the log information is randomized.
統計的ランダム化をログデータに導入するためにとられる具体的な手法にかかわらず、この手法は、追加のセキュリティおよびプライバシー保護を提供する。この匿名化されたログデータは、サーバシステムに送信される。上記のように、サーバ側では、システムは、複数のクライアントデバイスのログ情報をアグリゲートする可能性がある。このことの1つの利点は、アグリゲーションが統計的ランダム化を考慮しながら、訪問コンバージョンプロセスの信頼できる尺度を提供することである。これは、全体としてログデータに導入された偽陽性および偽陰性の数が相殺される傾向があるからである。したがって、システムが大きいデータセットにわたってログデータを評価するとき、ウェブサーバから検索結果を受信した後、クライアントデバイスが会社および他の関心のあるポイントを訪問する傾向があったかどうかを判断できる。 Regardless of the specific technique taken to introduce statistical randomization into log data, this technique provides additional security and privacy protection. This anonymized log data is sent to the server system. As mentioned above, on the server side, the system may aggregate log information from multiple client devices. One advantage of this is that aggregation provides a reliable measure of the visit conversion process, taking into account statistical randomization. This is because the number of false positives and false negatives introduced into the log data as a whole tends to be offset. Therefore, when the system evaluates log data across a large dataset, it can determine if the client device tended to visit the company and other points of interest after receiving the search results from the web server.
