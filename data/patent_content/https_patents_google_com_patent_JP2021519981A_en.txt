Abstract
本開示は、「読出し専用クッキー」(「ROC」)とも呼ばれることのある、セキュアで、ユーザに透過であり、かつ高効率のコンテンツプロバイダ固有識別子(「CPSID」)を提供する。これらのコンテンツプロバイダ固有識別子は、クライアントデバイスによって生成されて、コンテンツプロバイダの公開鍵で暗号化され得、それによって、サードパーティがマッチングを間接的に特定することが防がれるとともに、プロバイダ側のクッキーマッチングテーブルおよびリソース多用のトラッキング通信の必要性がなくなる。コンテンツプロバイダ固有識別子の生成は、順守サービス利用規約(ToS)が例えばドメイン内の所定のアドレスから取り出し可能なコンテンツプロバイダ、(例えばユーザが明示的に同意した)ホワイトリストに載っているコンテンツプロバイダ、および/または(例えばユーザが明示的に同意を拒否した)ブラックリストに載っていないコンテンツプロバイダについてのみ識別子が作成されるように、ユーザポリシーによって制御され得る。The present disclosure provides a secure, user-transparent, and highly efficient content provider-specific identifier (“CPSID”), sometimes referred to as a “read-only cookie” (“ROC”). These content provider unique identifiers can be generated by the client device and encrypted with the content provider's public key, which prevents third parties from indirectly identifying the match and the provider's cookie. Eliminates the need for matching table and resource-intensive tracking communications. The generation of content provider unique identifiers is for content providers whose Compliance Service Terms of Service (ToS) can be retrieved, for example, from a given address in the domain, whitelisted content providers (eg, the user has explicitly agreed), and / Or may be controlled by user policy so that identifiers are created only for content providers that are not blacklisted (eg, the user explicitly refused consent).
Description
本明細書は、ブラウザクッキーの置換えとしてのドメイン固有ブラウザ識別子に関する。 The present specification relates to domain-specific browser identifiers as a replacement for browser cookies.
ブラウザクッキー、またはデバイス固有、ユーザ固有、もしくはセッション固有の短い識別子は、ウェブページパブリッシャ、アドバタイザ、または他のそのようなエンティティなどのコンテンツプロバイダに対し、デバイス、ユーザ、またはセッションを一意に(または半一意に)識別するために使用される。これらのクッキーは、典型的には、コンテンツプロバイダによって、例えば最初のセッションまたはログインプロセスの間に生成され、クライアントデバイスのブラウザまたは他のアプリケーションからの後続の要求の中で提供される。次いで、パブリッシャは、各要求を同一識別子と(また暗黙的に同一のデバイス、ユーザ、またはセッションと)関連付けて、コンテンツ選択のカスタマイズを可能にすることができる。 A browser cookie, or a short device-specific, user-specific, or session-specific identifier, uniquely (or semi-) a device, user, or session to a content provider such as a web page publisher, advertiser, or other such entity. Used to identify (uniquely). These cookies are typically generated by the content provider, for example during the first session or login process, and provided in subsequent requests from the client device's browser or other application. Publishers can then associate each request with the same identifier (and implicitly with the same device, user, or session) to allow customization of content selection.
しかし、典型的なブラウザクッキーには、いくつかの重大な欠点がある。ウェブサイト内に埋込みコンテンツを提供するサードパーティコンテンツプロバイダは、ブラウザをトラッキングするためのクッキーを、ユーザが必ずしもこれらのサードパーティによって識別またはトラッキングされていることに気付いているとは限らない状態で提供することができる(なぜなら、例えばブラウザアドレスバー内に示されたアドレスは、メインページのソースを特定するにすぎず、必ずしも埋込みコンテンツのソースを特定するとは限らない場合があるからである)。したがって、ユーザが潜在的なプライバシの問題を特定し、個人情報のプライバシを制御することは、困難となり得る。 However, typical browser cookies have some serious drawbacks. Third-party content providers that provide embedded content within a website provide cookies for browser tracking without the user being aware that they are necessarily identified or tracked by these third parties. (Because, for example, the address shown in the browser address bar only identifies the source of the main page and may not necessarily identify the source of the embedded content). Therefore, it can be difficult for users to identify potential privacy issues and control the privacy of personal information.
加えて、典型的なブラウザクッキーは、ドメイン固有である。何らかのセキュリティ手段を提供しようとする試みの中で、ブラウザおよび他のアプリケーションのメーカーは、あるコンテンツプロバイダが、他のドメインの他のコンテンツプロバイダによって生成されたクッキーを読み出す、または受信することのないようにしている。これに対処するため、正当と不当の両方の理由で、コンテンツプロバイダは、クッキーマッチングを利用して、あるコンテンツプロバイダからのクッキーと別のコンテンツプロバイダのクッキーとの間の関連付けを生成する。クッキーマッチングの一実装形態は、ページ内の小さな埋込みコンテンツアイテム(例えば、ブラウザに、対応するクッキーの送信とともに画像を要求させる、「トラッキングピクセル」と呼ばれる1ピクセル×1ピクセルの「不可視の」画像)を利用するものであり、この要求は、あるエンティティによって受信され、別のエンティティにリダイレクトされる。この結果、そのピクセルを「レンダリングする」ための2回のラウンドトリップ通信交換、すなわち、第1のエンティティへの要求とリダイレクションアドレスによる応答、および第2のエンティティへの要求と応答が生じる。10個以上の埋込みトラッキングピクセルのある典型的なページでは、これらの通信の実施が大幅に時間をとることがある。通信が完了する前にユーザがそのページから離れてナビゲートする場合、マッチングは特定されないことがある。結果として、クッキーマッチングは遅く、しばしば未完了であり、ウェブページのレンダリングの速度を遅くし、帯域幅を消費し、バッテリ寿命を縮める。さらに、クッキーマッチングは、サードパーティエンティティがオンライン行動を非透過式に、ユーザが明示的に同意することなくトラッキングすることを可能にし、これはユーザのプライバシにとっての脅威となる。 In addition, typical browser cookies are domain-specific. In an attempt to provide some form of security, browser and other application manufacturers ensure that one content provider does not read or receive cookies generated by other content providers in other domains. I have to. To address this, for both legitimate and unjustified reasons, content providers utilize cookie matching to generate an association between a cookie from one content provider and a cookie from another content provider. One implementation of cookie matching is a small embedded content item in a page (for example, a 1 pixel x 1 pixel "invisible" image called a "tracking pixel" that causes the browser to request an image with the transmission of the corresponding cookie). This request is received by one entity and redirected to another. This results in two round trip communication exchanges to "render" the pixel: a request to the first entity and a response by the redirect address, and a request and response to the second entity. For a typical page with 10 or more embedded tracking pixels, these communications can take a significant amount of time. Matching may not be specified if the user navigates away from the page before the communication is complete. As a result, cookie matching is slow, often incomplete, slows down web page rendering, consumes bandwidth, and shortens battery life. In addition, cookie matching allows third-party entities to non-transparently track online behavior without the explicit consent of the user, which poses a threat to the user's privacy.
本明細書において論じるシステムおよび方法は、従来のブラウザクッキーを、「読出し専用クッキー(read-only cookie)」(「ROC」)とも呼ばれることのある、セキュアで、ユーザに透過であり、かつ高効率のコンテンツプロバイダ固有識別子(content provider-specific identifier)(「CPSID」)と置き換える。これらのコンテンツプロバイダ固有識別子は、クライアントデバイスによって生成されて、コンテンツプロバイダの公開鍵で暗号化され得、それによって、サードパーティがマッチングを間接的に特定することが防がれるとともに、プロバイダ側のクッキーマッチングテーブル、ならびにユーザへの透過性とクッキーの使われ方の明確な把握および監査とを妨げるリソース多用のクッキーマッチング通信およびシステムの必要性がなくなる。コンテンツプロバイダ固有識別子の生成は、順守サービス利用規約(compliant terms of service)(ToS)が例えばドメイン内の所定のアドレスから取り出し可能なコンテンツプロバイダ、(例えばユーザが明示的に同意した)ホワイトリストに載っているコンテンツプロバイダ、および/または(例えばユーザが明示的に同意を拒否した)ブラックリストに載っていないコンテンツプロバイダについてのみ識別子が作成されるように、ユーザポリシーによって制御され得る。 The systems and methods discussed herein refer to traditional browser cookies as secure, user-transparent, and highly efficient, sometimes referred to as "read-only cookies" ("ROCs"). Replace with the content provider-specific identifier (“CPSID”) of. These content provider unique identifiers can be generated by the client device and encrypted with the content provider's public key, which prevents third parties from indirectly identifying the match and the provider's cookie. Eliminates the need for matching tables, as well as resource-intensive cookie matching communications and systems that impede user transparency and clear understanding and auditing of cookie usage. The generation of a content provider unique identifier is listed on a content provider whose compliant terms of service (ToS) can be retrieved, for example from a given address in the domain, (eg, the user has explicitly agreed). User policies may control that identifiers are created only for content providers that are and / or content providers that are not blacklisted (eg, the user has explicitly denied consent).
コンテンツプロバイダ固有識別子は、入力であるブラウザ固有、アプリケーション固有、デバイス固有、またはセッション固有のプライベート識別子(例えば、デバイスの他のユーザとは別の、特定のユーザのブラウザのインスタンスに固有の識別子)およびコンテンツプロバイダのドメイン名またはアドレス(例えば「example.com」)から、暗号学的ハッシュ(例えば、MD5、SHA256などのような任意の適切なハッシングアルゴリズムなど)として生成され得る。いくつかの実装形態では、追加のソルトを与えるために、ハッシュ入力は、コンテンツプロバイダ固有識別子の作成時間または期限満了時間を含むことができ、これは、例えばどちらか一方の入力値と連結され得る。ハッシュ入力の複雑さを同様に増大させるために、デバイスのタイプもしくはシリアル番号、クロックスキュー時間、ハードウェア識別子、または他の任意のそのようなデータなど、他のエントロピーソースを利用することができる。いくつかの実装形態では、「期限満了時間」は、コンテンツプロバイダ固有識別子が期限満了するまでの週間の数、月数、または年数など、粒度による値(granular value)とすることができ、それによって、クライアントによる自動再生成が可能となる。 Content provider-specific identifiers are input browser-specific, application-specific, device-specific, or session-specific private identifiers (for example, identifiers specific to a particular user's browser instance, separate from other users of the device) and It can be generated from the content provider's domain name or address (eg "example.com") as a cryptographic hash (eg any suitable hashing algorithm such as MD5, SHA256, etc.). In some implementations, the hash input can include the content provider unique identifier creation time or expiration time to give additional salt, which can be concatenated with, for example, one of the input values. .. Other entropy sources can be utilized, such as device type or serial number, clock skew time, hardware identifier, or any other such data, to increase the complexity of the hash input as well. In some implementations, the "expiration time" can be a granular value, such as the number of weeks, months, or years before the content provider's unique identifier expires. , Automatic regeneration by the client is possible.
他の実装形態では、コンテンツプロバイダ固有識別子を、ルックアップテーブル内にコンテンツプロバイダのドメイン(例えばexample.com)をキーとして記憶された、ランダムに生成された値とすることができる。これにより、必要となるプロセッサリソースは暗号学的ハッシュ結果を計算するよりも少なくなり得るが、テーブルのための大きな記憶空間が必要となり得る。 In other implementations, the content provider unique identifier can be a randomly generated value stored in the lookup table with the content provider's domain (eg, example.com) as the key. This can require less processor resources than computing cryptographic hash results, but can require large storage space for tables.
コンテンツプロバイダ固有識別子は、暗号化された読出し専用クッキー(「eROC」)を生成するために、クライアントデバイスによってコンテンツプロバイダの公開鍵を通じて暗号化され得る。いくつかの実装形態では、クライアントデバイスは、結果として得られるeROCが単一ドメインのものであっても大いに異なるように、暗号化の前にコンテンツプロバイダ固有識別子を作成タイムスタンプおよび/または乱数(例えばナンスソルト)と連結することができる。したがって、意図されたコンテンツプロバイダは、コンテンツプロバイダ固有識別子を復号および抽出できる可能性があり、一方、他のコンテンツプロバイダ(例えばクッキーマッチングを実施しようと試みるコンテンツプロバイダ)は、そうすることができない。eROCをクライアントデバイスから中継体を経て意図されたコンテンツプロバイダに送信する間に保護するために、作成タイムスタンプおよび/またはナンス値を使用して、意図されたコンテンツプロバイダ以外の誰にも無用または解読不能な不安定なeROC(unstable eROC)を生成することができる。コンテンツプロバイダ間のデータ交換を可能にするために、第1のコンテンツプロバイダが、第2のコンテンツプロバイダのためにクライアントデバイスにeROCを要求することができ、受信したeROCを第2のコンテンツプロバイダに転送することができる。受信したクッキーは第2のコンテンツプロバイダの公開鍵で暗号化されているので、第1のコンテンツプロバイダは、第2のコンテンツプロバイダのROCを復号または回復することはできない可能性があるが、その代わりに単にeROCを転送し、関連するデータを要求することができる。したがって、適切なeROCを所有しているということは、データ交換についてクライアントデバイスによる認可がなされたことを意味する。各要求がクライアントデバイスを通過しなければならないので、クライアントデバイスは、プライバシおよびコンテンツプロバイダ間のデータ交換を制御することができ、またそのような任意の転送について監査ログまたはデータ交換ログを生成することもでき、それによって、ユーザに対する透過性が増大する。 The content provider unique identifier can be encrypted by the client device through the content provider's public key to generate an encrypted read-only cookie (“eROC”). In some implementations, the client device creates a content provider-specific identifier prior to encryption so that the resulting eROC is very different even if it is from a single domain. One implementation of cookie matching is a small embedded content item in a page (for example, a 1 pixel x 1 pixel "invisible" image called a "tracking pixel" that causes the browser to request an image with the transmission of the corresponding cookie). This request is received by one entity and redirected to another. This results in two round trip communication exchanges to "render" the pixel: a request to the first entity and a response by the redirect address, and a request and response to the second entity. For a typical page with 10 or more embedded tracking pixels, these communications can take a significant amount of time. Matching may not be specified if the user navigates away from the page before the communication is complete. As a result, cookie matching is slow, often incomplete, slows down web page rendering, consumes bandwidth, and shortens battery life. In addition, cookie matching allows third-party entities to non-transparently track online behavior without the explicit consent of the user, which poses a threat to the user's privacy. In addition, after the cookie matching table has been filled, the relay server and content server can then freely share user data without any transparency or user control to the user, because the server Further requests and data exchanges between them do not have to go through the client device. Such user data sharing may continue even if the user clears the cached cookie or account identifier on the client device. Even if you use the opt-out mechanism on the server side, you cannot ensure security or privacy.
